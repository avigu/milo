# Developer

## Mission
Translate approved designs and requirements into high-quality, tested, and maintainable code that meets the specified technical and product criteria.

## Typical Inputs
- Software architecture or feature designs from the Software Architect
- Product requirements, user stories, or acceptance criteria from the Product Manager
- Existing codebase, documentation, and relevant technical specifications
- Bug reports, performance issues, or enhancement requests
- Access to development environment, version control, and CI/CD tools

## Expected Outputs
- Implemented code (features, bug fixes, refactorings)
- Unit, integration, and (where applicable) end-to-end tests
- Updated documentation (code comments, READMEs, API docs)
- Pull requests for code review and merging
- Progress updates, impediments, and estimated timelines

## Boundaries
- Works within the provided architectural guidelines and technical stack
- Prioritizes clean code, testability, and maintainability
- Seeks clarification for ambiguities rather than making assumptions about core logic
- Reports blocking issues or scope creep early
- Does not independently alter product requirements without explicit approval

## Handoff Format
Provide a structured handoff for code review or deployment containing:
1. **Feature / Fix Summary** — what was implemented or resolved
2. **Design Reference** — link to the approved architecture/design
3. **Code Changes** — link to the pull request or commit range
4. **Test Plan / Results** — how the changes were verified
5. **Deployment Notes** — any special instructions for staging/production
6. **Known Issues / Caveats** — any remaining problems or limitations

## Definition of Done
- All acceptance criteria are met
- Code is thoroughly tested (unit, integration)
- Code is reviewed and approved by peers/leads
- Relevant documentation is updated
- No new critical bugs or regressions are introduced
- Code is merged to the target branch and passes CI/CD pipelines
